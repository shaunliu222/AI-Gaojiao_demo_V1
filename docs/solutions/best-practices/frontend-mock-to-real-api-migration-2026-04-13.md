# Frontend Mock-to-Real API Migration Pattern

## Context

After Phase 3 backend was complete (61 Java files, 30+ endpoints), the frontend still imported all data from `mocks/data.ts`. Additionally, the backend required PostgreSQL + Redis (Docker), making local testing impossible without infrastructure. This learning documents the pattern for migrating frontend pages from mock to real API and enabling Docker-free backend startup.

## Guidance

### 1. H2 Dev Profile for Docker-Free Backend

Create separate H2-compatible SQL files instead of making PostgreSQL init.sql dual-compatible:

- `schema-h2.sql`: `BIGSERIAL` → `BIGINT GENERATED BY DEFAULT AS IDENTITY`, `JSONB` → `TEXT`, no FK constraints
- `data-h2.sql`: `ON CONFLICT` → `MERGE INTO ... KEY(column)`, one statement per row
- `application-dev.yml`: set `spring.sql.init.schema-locations` + `data-locations`
- Maven profile `no-redis`: exclude `sa-token-redis-jackson` and `spring-boot-starter-data-redis` so Sa-Token falls back to in-memory storage

Startup command: `mvn spring-boot:run -Dspring-boot.run.profiles=dev -P no-redis`

Key gotcha: BCrypt hashes must be generated by the actual `BCryptPasswordEncoder`, not copied from documentation. Use a temp main class to generate real hashes.

### 2. Centralized API Service Layer

Add all API functions to one file (`services/request.ts`) grouped by module, following the existing `authApi` pattern:

```typescript
export const userApi = {
  list: (params?) => request.get('/api/users', { params }),
  create: (data) => request.post('/api/users', data),
  // ...
};
```

For SSE streaming, use raw `fetch` + `ReadableStream` (not axios — it doesn't support streaming):

```typescript
chatStream: async (data, onData, onDone, onError) => {
  const response = await fetch('/api/ai/chat/stream', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
    body: JSON.stringify({ ...data, stream: true }),
  });
  const reader = response.body?.getReader();
  // parse SSE lines: data: {json}\n
};
```

### 3. Page Migration Pattern

For each page, the migration is:
1. Remove `import { mockXxx } from '@/mocks/data'`
2. Add `useState` + `useEffect` + `useCallback` for data fetching with loading state
3. Replace hardcoded data with API calls
4. Add `message.error()` on catch blocks
5. Add `Popconfirm` for destructive operations
6. Wire pagination to backend `PageResult` format `{ total, list, page, size }`

Keep `mocks/data.ts` — non-migrated pages still need it.

### 4. Graceful Degradation for Chat

The chat page has triple fallback: backend proxy → direct OpenClaw → mock. This ensures the page works regardless of infrastructure state:

```typescript
// 1. Try backend proxy /api/ai/chat/stream
// 2. If backend unavailable, try direct /gateway/v1/chat/completions
// 3. If both fail, use mockStreamResponse
```

## Why This Matters

- Without H2 dev profile, developers need Docker Desktop running just to test a login page
- Without centralized API layer, each page reinvents HTTP calls and error handling
- Without mock fallback, the chat page is completely broken when OpenClaw isn't running
- The migration pattern (mock → real with preserved mock file) allows incremental integration across Phases

## When to Apply

- Phase 4/5/6 frontend integration: agents, skills, plugins, resources, knowledge graph pages
- Any new page that starts with mock data
- When adding a new backend module that needs frontend CRUD

## Examples

### Before (mock)
```tsx
import { mockUsers } from '@/mocks/data';
const [users] = useState(mockUsers);
```

### After (real API)
```tsx
import { userApi } from '@/services/request';
const [users, setUsers] = useState([]);
const fetchUsers = useCallback(async () => {
  const res = await userApi.list({ page, size: 20, keyword });
  setUsers(res.data?.list || []);
}, [page, keyword]);
useEffect(() => { fetchUsers(); }, [fetchUsers]);
```
