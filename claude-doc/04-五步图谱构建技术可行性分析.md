# 五步渐进式图谱构建 — 技术可行性分析

## 目标

在开发 Phase 4/5 之前，逐步验证"五步渐进式图谱构建"所需的每个技术环节是否具备、是否有成熟开源方案。

---

## 逐步分析

### Step 1: 骨架定义（专家定义 或 LLM 辅助提取框架）

| 组件 | 需求 | 现状 | 就绪度 |
|------|------|------|--------|
| 前端图谱编辑器 | AntV G6 拖拽创建节点/关系 | Phase 2 已实现基础 G6 可视化（知识图谱页面） | ✅ 90% |
| LLM 框架提取 | 上传资料 → LLM 分析 → 输出章/节/知识点骨架 | OpenClaw Agent + Function Calling 可实现 | ✅ 可用 |
| 图数据库存储 | Neo4j 存骨架节点和关系 | **未集成** — docker-compose 无 Neo4j | ❌ 需新增 |

**Gap**: Neo4j 需要加入 docker-compose 并集成 Spring Data Neo4j。

---

### Step 2: 实体/关系抽取（MinerU 解析 → LLM 抽取）

| 组件 | 需求 | 现状 | 就绪度 |
|------|------|------|--------|
| **MinerU** 文档解析 | PDF/PPT/Word → Markdown | MinerU v3.0.5 已发布，支持 Docker 部署 + REST API (`POST /tasks`)，GPU 加速 | ✅ 成熟可用 |
| LLM 实体抽取 | Markdown → 实体(知识点/概念/公式) + 关系(前置/包含/关联) | 需设计 Prompt，可参考 GraphRAG 的抽取 Prompt + Neo4j LLM KG Builder | ⚠️ 需自研 Prompt |
| Embedding 模型 | BGE-M3 中英文 Embedding | OpenClaw 有完整 Embedding 能力（`src/memory-host-sdk/host/embeddings-*.ts`），支持 OpenAI/Ollama/Voyage/Gemini/Bedrock/Mistral 等 | ✅ 可复用 |
| 向量数据库 | Milvus 存储知识片段向量 | **未集成** — docker-compose 无 Milvus | ❌ 需新增 |

**Gap**:
1. MinerU 需部署为 Docker 服务（需 GPU，8GB+ VRAM）
2. Milvus 需加入 docker-compose
3. LLM 实体抽取 Prompt 需设计（可参考 GraphRAG/Neo4j KG Builder）

| Milvus 向量检索 | 语义检索挂载的 Markdown 片段 Top-K | Spring AI + Milvus VectorStore 或 Milvus Java SDK | ✅ 技术成熟 |

| LLM 问答生成 | 知识片段 + 图谱上下文 → Prompt → AI 回答 | OpenClaw Agent 可实现（Function Calling + 自定义 Tool） | ✅ 可复用 |
| 引用溯源 | 回答中标注知识来源和图谱路径 | 需在 Prompt 中要求 LLM 输出引用标记 | ⚠️ Prompt 设计 |

**Gap**: 混合检索策略和引用溯源 Prompt 需设计。

## 关键阻塞项（必须在 Phase 5 前解决）

### 阻塞 1: Neo4j 集成
- **方案**: docker-compose 新增 Neo4j 5 服务 + `spring-boot-starter-data-neo4j` 依赖
- **参考**: [Spring Data Neo4j](https://neo4j.com/docs/getting-started/languages-guides/java/spring-data-neo4j/), [Spring Boot + Neo4j 知识图谱教程](https://jxausea.medium.com/spring-boot-integrates-neo4j-to-implement-a-simple-knowledge-graph-fca16db05ead)
- **工作量**: ~2 天（docker 配置 + Entity/Repository + Cypher API）

### 阻塞 2: Milvus 集成
- **方案**: docker-compose 新增 Milvus 2.4 + Spring AI Milvus VectorStore 或 `milvus-sdk-java`
- **参考**: [Spring AI + Milvus](https://docs.spring.io/spring-ai/reference/api/vectordbs/milvus.html), [Milvus Java SDK](https://github.com/milvus-io/milvus-sdk-java)
- **注意**: Milvus standalone 需 etcd + MinIO（已有） 作为依赖，docker-compose 会较复杂
- **替代方案**: pgvector（PostgreSQL 向量扩展）— 如果规模不大（<100 万向量），pgvector 更轻量
- **工作量**: ~2 天

### 阻塞 3: MinerU 部署
- **方案**: Docker 部署 `mineru-api`，提供 `POST /tasks` REST API
- **参考**: [MinerU Docker 部署](https://opendatalab.github.io/MinerU/quick_start/docker_deployment/)
- **硬件要求**: GPU（Volta+, 8GB+ VRAM），如无 GPU 可用 CPU 模式（速度慢）
- **替代方案**: 如果无 GPU，可用 Apache Tika + pdf.js 做基础解析（质量低于 MinerU）
- **工作量**: ~1 天（Docker 部署 + REST API 对接）

### 阻塞 4: LLM Prompt 设计
- **实体抽取 Prompt**: 参考 [GraphRAG](https://github.com/microsoft/graphrag) 的 entity extraction prompt + [Neo4j LLM KG Builder](https://neo4j.com/blog/developer/llm-knowledge-graph-builder-release/)
- **智能切片 Prompt**: 需自研，核心思路是将图谱节点列表作为上下文输入 LLM，让 LLM 决定文本片段应挂载到哪个节点
- **工作量**: ~3 天（设计 + 测试 + 迭代）

## 用户确认的技术决策

1. **无 GPU** — MinerU 需要用 CPU 模式或替代方案（Apache Tika / marker）
2. **向量数据库: pgvector 优先** — Demo 阶段用 pgvector（PostgreSQL 扩展），后期迁移 Milvus

## 调整后的方案

### MinerU 替代方案（无 GPU）

| 方案 | 质量 | 速度 | 部署难度 |
|------|------|------|---------|
| MinerU CPU 模式 | 高（但慢） | 慢（10x） | Docker 部署 |
| marker（Python） | 中高 | 较快 | pip install |
| Apache Tika + pdf.js | 中 | 快 | Java 原生集成 |
| LLM 直接解析（智谱 GLM-4V） | 高 | 依赖 API | 零部署，调 API |

**推荐**: Demo 阶段用 **LLM 直接解析**（上传 PDF → 调 GLM-4V/GPT-4o 多模态 API → 输出 Markdown）。零基础设施依赖，通过 OpenClaw 现有 Provider 即可调用。后期有 GPU 再接 MinerU。

### pgvector 替代 Milvus

PostgreSQL 16 已支持 pgvector 扩展。只需在 init.sql 中 `CREATE EXTENSION vector;`，无需额外服务。Spring Boot 通过 JdbcTemplate 或 Spring AI PgVectorStore 即可操作。
